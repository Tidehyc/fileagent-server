# FileAgent Server — 需求文档优化总结

> 基于 v1.0 需求文档的优化建议和实施修改记录

---

## 一、已完成修改

### 🔴 高优先级（3项）

| 优化项 | 涉及章节 | 改动内容 |
|--------|---------|---------|
| 心跳保活锁粒度 | `3.2.4` | 心跳检查改为三段式：锁内取指针 → 释放锁 → 无锁 ping → 锁内归还，避免 `mysql_ping` IO 操作独占互斥锁；同时增加缩容策略，低谷期自动回收超出 `min_size` 的空闲连接 |
| 事务管理 | `3.3.2` | 新增 `TransactionGuard` RAII 类；`ConnectionPool` 增加 `allocateConnection/returnConnection` 原始接口；DAO 方法增加 `MYSQL* conn` 重载，支持跨 DAO 事务 |
| MD5 → SHA-256 | 全文 11 处 | 秒传流程、files 表、chunks 表、API 接口名、Crypto 注释、依赖说明、评估清单等所有 MD5 引用 |

### 🟡 中优先级（3项）

| 优化项 | 涉及章节 | 改动内容 |
|--------|---------|---------|
| 向量存储持久化 | `3.6.2` | `VectorStore` 新增 `saveToDisk/loadFromDisk` 方法 + `persist_path_` 字段；RAG 流程图标注异步处理 + 定期持久化，避免服务重启后全量重建索引 |
| 分片传输格式 | `5.2` | 分片上传从 `JSON + Base64` 改为 `multipart/form-data + 二进制`，消除 33% 编码开销，与普通文件上传接口保持一致 |
| 密码 salt 字段 | `4.2` | `users` 表删除冗余 `salt` 字段，bcrypt 的 salt 已内嵌在哈希结果（`$2b$10$...`）中，无需单独字段存储 |

### 🚀 HTTP Server 实现方向调整（全文多处）

| 项 | 内容 |
|----|------|
| 推荐方案 | **Drogon 框架** — 底层 epoll ET + 多线程 Reactor，与原始设计一致，省去手写 HTTP 状态机的脏活 |
| 备选方案 | epoll 自实现 + **llhttp** 解析器（保留手写网络层，HTTP 解析交给专业库） |
| 已改范围 | `3.1` 节重写、`1.2/1.3` 技术栈更新、目录结构精简、`vcpkg.json` 加 `drogon`、`CMakeLists.txt` 改为 Drogon 构建、`main.cpp` 更新为 Drogon 入口、`config.yaml` 端口改为 `10086` |

## 二、中低优先级（未修改，仅记录）

以下问题已识别但本次未纳入修改：

| 优先级 | 问题 | 说明 |
|--------|------|------|
| 🟡 | 缺少文件重命名/移动接口 | Agent 的 `organize_files_by_type` 工具依赖此接口 |
| 🟡 | 缺少文件类型白名单校验 | 上传文件应限制类型 |
| 🟢 | 分享码过长（VARCHAR 32） | 推荐 6-8 位 base62 随机码 |
| 🟢 | Session 表缺少过期清理 | 需要定时任务清理过期 session |
| 🟢 | Agent `delete_file` 缺乏确认机制 | 两阶段删除或确认令牌机制 |
| 🟢 | Docker 镜像可改用 distroless | ubuntu:22.04 → distroless 可大幅减体积 |
| 🟢 | 缺少健康检查配置 | docker-compose 中应加 healthcheck |
| 🟢 | 日志缺少持久化卷 | docker-compose volumes 缺少日志 |

## 三、核心原则：手写 vs 框架

### 手写有区分度的（核心竞争力）

| 组件 | 手写原因 |
|------|---------|
| MySQL 连接池 | 展现并发编程、RAII、生产者消费者模型能力 |
| DAO + 事务管理 | 展现数据库抽象设计能力 |
| AI Agent (ReAct) | 展现 LLM 编排、工具系统设计能力 |
| RAG 管道 | 展现向量检索、文本处理能力 |
| llama.cpp 集成 | 展现 C++ 与 ML 推理引擎集成能力 |
| LRU Cache | 展现数据结构与算法能力 |

### 交给框架的（非核心劳动）

| 组件 | 替代方案 | 原因 |
|------|---------|------|
| HTTP 状态机解析 | Drogon 内置 | 繁琐且无技术区分度 |
| Socket + epoll 事件循环 | Drogon 底层实现 | 理解原理即可，手写成本高 |
| 路由注册 | Drogon `registerHandler` | 框架特性，语义清晰 |
| 中间件链 | Drogon `registerMiddleware` | 框架特性 |
